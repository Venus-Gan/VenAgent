# Phase1.模块映射

## 目的
把现有 `AGI-saber` 的能力，按 `VenAgent` 的目标架构重新归类。

这份映射的目标不是重写实现，而是先回答三个问题：
- 哪些能力必须保留在底层
- 哪些能力应该封装成 MCP tool
- 哪些能力应该交给 `LangGraph` 和 `agentteam`

---

## 目标分层

### 1. Core
保留原始业务能力和内部数据流转逻辑。

### 2. MCP
把 `Core` 能力对外暴露成标准化工具。

### 3. Graph
负责任务路由、状态流转、子流程调度。

### 4. AgentTeam
负责可扩展子 agent 的注册、选择、编排和预制能力。

---

## 现有能力映射

### A. 工具能力
#### 现有场景
- 时间查询
- 天气查询
- Web 搜索
- 命令执行
- 文档相关工具
- 外部工具注册

#### 目标归属
- `MCP tool`

#### 处理方式
- 保留内部实现
- 统一封装为 tool
- 让上层只看 schema，不直接碰实现细节
- `search_web` 的网页获取后端改为 `fetch MCP`，不再保留 Tavily 配置位
- 沙箱本身不并入 tool 列表，作为独立执行边界保留

---

### B. RAG 能力
#### 现有场景
- 检索
- 向量召回
- 混合搜索
- 文档问答
- 结果合成

#### 目标归属
- `Core` + `MCP`

#### 处理方式
- 核心检索逻辑保留
- 暴露 `rag.search`、`rag.retrieve`
- 预制 agent 调用 MCP 层完成检索

---

### C. Memory 能力
#### 现有场景
- 会话记忆
- 长期记忆
- 状态保存
- 记忆读取

#### 目标归属
- `Core` + `MCP`

#### 处理方式
- 保留底层记忆存储与读写逻辑
- 暴露 `memory.read`、`memory.write`
- 由 Graph 或 AgentTeam 统一消费

---

### D. Document 能力
#### 现有场景
- 上传
- 解析
- 清洗
- 切片
- 入库

#### 目标归属
- `Core` + `MCP`

#### 处理方式
- 核心解析管线保留
- 暴露 `document.parse`、`document.ingest`
- 后续可以替换解析引擎，但不改外部契约

---

### E. Agent 编排能力
#### 现有场景
- 路由
- 选择 agent
- 多步任务
- 结果汇总

#### 目标归属
- `LangGraph`

#### 处理方式
- 用 graph 统一接管流程
- 不再把编排逻辑散落在业务函数里
- agent 选择改为状态驱动

---

### F. 原固定四个 RAG Agent
#### 现有场景
- 原来写死在 RAG 里的四个 agent

#### 目标归属
- `agentteam/presets`

#### 处理方式
- 作为预制子 agent 迁入
- 维持原有职责，但不再硬编码在 RAG 内部
- 后续任务可以按需复用

---

### G. 沙箱能力
#### 现有场景
- 命令执行隔离
- 资源限制
- 执行安全

#### 目标归属
- `Sandbox` 独立模块

#### 处理方式
- 不并入 MCP tool 主集合
- 作为受控执行边界独立维护
- 由工具层按权限调用

---

### H. 知识图谱能力
#### 现有场景
- KGStore
- GraphMemory
- Neo4j 联动

#### 目标归属
- `KnowledgeGraph` 独立模块

#### 处理方式
- 作为独立记忆子系统保留
- 与 RAG / memory 解耦命名
- 可被恢复逻辑和 agentteam 调用

---

### I. 恢复 / 快照 / 取消
#### 现有场景
- 启动恢复
- 任务快照
- 中断取消

#### 目标归属
- `RuntimeGovernance` 独立模块

#### 处理方式
- restore / snapshot / cancel 作为运行时治理
- LangGraph checkpoint 可以接管一部分执行态恢复
- 业务快照与取消注册表继续保留

---

### J. HTTP API / 前端交互层
#### 现有场景
- HTTP 路由
- 前端页面
- 用户交互层

#### 目标归属
- `API` / `Frontend`

#### 处理方式
- Phase 1 暂不修改
- 后续单独规划，不影响当前重构主线

---

## 保留 / 重构 / 新增清单

### 保留
- RAG 核心检索逻辑
- memory 核心存取逻辑
- 文档解析底座
- 现有业务规则

### 重构
- 工具暴露方式
- agent 编排方式
- 调用链和依赖关系
- 任务路由方式

### 新增
- MCP 包装层
- LangGraph 图定义
- `agentteam` 模块
- 统一状态协议

---

## 迁移优先级

### P0
- RAG tool 包装
- memory tool 包装
- `agentteam` 目录与注册机制

### P1
- LangGraph 主入口
- 预制 agent 迁移
- 路由逻辑切换

### P2
- 文档 ingest tool 化
- 外部工具全量 MCP 化
- 旧调用点逐步清理

---

## 验收标准

当以下条件满足时，模块映射就算完成：
- 每个现有能力都有明确归属
- `Core / MCP / Graph / AgentTeam` 边界清晰
- 原固定 RAG agent 有落点
- 后续实现不会因为职责不清而返工
