# Phase1.实施路线与风险

## 目的
把 Phase 1 的推进顺序、预制子 agent 规划和风险控制单独抽出来，方便后续按步骤执行而不是一次性推翻。

---

## 执行步骤
| 步骤 | 输出物 | 验收标准 |
|---|---|---|
| Step 1：锁定分层 | 分层说明、模块职责表、保留与重构清单 | 每个能力都能明确归属 |
| Step 2：映射现有模块 | 模块映射表、MCP 化清单、保留核心清单 | 没有未定义归属的关键模块 |
| Step 3：定义 `agentteam` | 配置协议、注册机制、预制列表、自定义扩展接口 | 原四个 RAG agent 能无歧义迁入 presets |
| Step 4：定义 MCP 包装方式 | tool 命名、输入输出 schema、权限边界 | LangGraph 和 agentteam 只依赖工具契约 |
| Step 5：定义迁移顺序 | 迁移顺序和停靠点 | 每一步都可独立停止并验证稳定性 |

---

## 预制子 agent 规划
| 预制 agent | 场景 | 说明 |
|---|---|---|
| `research` | 资料探索与信息归纳 | 做初步结论 |
| `doc_qa` | 文档问答与上下文检索 | 做证据回溯 |
| `synthesis` | 多来源融合与结果整理 | 输出统一口径 |
| `ops` | 执行类、辅助类、工具类协作 | 做任务型操作 |

## 后续扩展方式
| 方式 | 要求 |
|---|---|
| 新增自定义 agent | 放到 `agentteam/custom`，再注册接入 |
| 挂接到图中 | 由 LangGraph 节点或子图调用 |
| 约束 | 不绕过 MCP，不复制核心检索逻辑，不把流程控制塞回 RAG |

---

## 风险控制
| 风险 | 控制方式 |
|---|---|
| RAG / memory 重写过度 | 只包一层，不推倒重写 |
| 工具层和编排层耦合 | 所有外部能力先统一经过 MCP |
| agentteam 变成新一层工具堆砌 | 只放子 agent 定义、注册、运行契约 |
| 一次性改动太大 | Phase 1 只定边界和结构，替换分阶段推进 |

---

## 交付物
| 交付物 | 内容 |
|---|---|
| 分层方案 | Core / MCP / Graph / AgentTeam |
| 模块映射表 | 现有模块到目标架构的映射 |
| agentteam 设计 | 子 agent 体系和扩展接口 |
| MCP 封装原则 | RAG / memory / document 的统一工具边界 |
| 迁移顺序 | 逐步切换的实施路径 |

---

## 完成标准
- 架构边界已经固定
- `agentteam` 的定位已经明确
- RAG / memory 作为底层资产被保留
- 对外工具接口统一为 MCP
- 后续实现可以按模块逐步推进
