# VenAgent 重构 — 产品需求文档

> 本文档从已有的技术审计报告和 Phase 1 设计文档逆向推导得出。
> 它不是对"怎么做"的描述，而是对"要什么"的回答。

---

## 一、背景与动机

VenAgent 的前身是一个已具备完整能力的 AI Agent 应用——它能聊天、能做知识检索（RAG）、能调用工具、能执行多步任务（ReAct），还有记忆系统、文档管理、沙箱执行和子代理协作。

问题是，这些能力是**堆砌**出来的，不是**设计**出来的。

当前系统的"大脑"（UnifiedAgent）承担了太多角色：它既要负责理解用户意图、又要自己调用工具、还要管理记忆、协调子代理、保存快照、处理恢复——就像一个工厂里，厂长、调度员、质检员、仓库管理员都是同一个人。结果是，任何一处改动都可能牵动全身，新能力加不进去，旧逻辑不敢删。

同时，系统对外部工具的依赖是"硬连线"的——搜索绑定了特定服务商，工具调用直接穿透到底层实现。一旦要换一个搜索后端，或者要增加一个新的外部能力，就需要在主流程里"接线"，风险高、成本大。

更紧迫的是，当前配置文件里存在**硬编码的密钥和密码**，这是一个必须在任何功能开发之前解决的阻塞性问题。

这次重构的目标，不是给系统加新功能，而是**把已有的能力重新整理清楚**——让每个模块有明确的职责边界，让外部能力通过标准接口接入，让子代理可以灵活扩展而不是写死在代码里，同时保障迁移过程中用户看到的服务不受影响。

---

## 二、目标用户

| 用户角色 | 核心诉求 |
|---------|---------|
| **现有用户** | 重构后原先的功能继续可用（聊天、RAG、工具调用、文档处理），体验不降级 |
| **开发者** | 能在一个边界清晰、模块职责明确的代码库上添加新能力或替换后端 |
| **运维者** | 配置安全可控，密钥不泄露到仓库，服务启动和降级行为可预期 |

---

## 三、核心目标

### 目标一：职责分家

当前 UnifiedAgent 一个人干了太多活。重构后：

- 系统需要一个**专门的意图理解层**，负责在收到用户请求后，判断这次应该走哪条路径（简单聊天？检索知识库？调用某个工具？还是执行一个多步任务？），并输出一个结构化的决策，而不是在代码里靠关键词硬猜。
- 系统需要一个**专门的工具边界层**，所有外部能力（搜索、文档解析、RAG 检索、记忆读写等）都通过统一的标准接口暴露给上层，上层只认接口，不关心后端是谁。
- 系统需要一个**专门的子代理层**，负责管理那些执行具体任务的子代理（比如做资料搜集的、做文档问答的、做结果汇总的），并支持在不改主流程的前提下新增或替换子代理。

### 目标二：接口标准化

当前系统的工具是"本地函数 + 手工降级链"拼出来的，搜索换了后端就要改核心代码。重构后：

- RAG、记忆、文档、外部工具全部通过 MCP（Model Context Protocol）标准接口暴露。
- 上层（编排引擎、子代理）只接触标准的输入输出定义，不直接依赖底层实现。
- 可以替换任何一个后端（比如把搜索从 A 服务商换成 B 服务商），而不影响调用方。

### 目标三：子代理可扩展

当前系统内置了四个固定的子代理（research、writer、review、doc），它们被写死在代码里。重构后：

- 这四个子代理作为"预制模板"保留，但不再硬编码在主流程中。
- 开发者可以新增自定义子代理，注册后即可被编排引擎发现和使用。
- 子代理的能力边界由配置和权限策略控制，不会越权访问底层能力。

### 目标四：编排引擎现代化

当前系统的多步任务执行（ReAct 模式）是基于自研的线程模型手写的——自己管理拓扑排序、并发控制、竞速逻辑、重试和取消。重构后：

- 采用成熟的编排框架（LangGraph）来管理任务的状态流转、路由分发、并行执行和中断恢复。
- 执行状态可保存、可回放、可恢复——长任务不会因为一次中断就丢失所有进度。
- 原先的四种运行模式（对话/检索/工具/多步任务）都能映射到新的编排方式中。

### 目标五：安全基线修复

- 配置文件中不得包含任何真实密钥、密码或 Token。
- 敏感信息通过环境变量或本机覆盖文件注入，且这些文件不提交到版本控制。
- 命令执行等高危操作有明确的权限边界和审计能力。

### 目标六：渐进迁移，可回滚

- 重构不追求一次性推倒重来，而是按模块逐步迁移。
- 每一步都有兼容层确保旧接口继续可用。
- 每一步都有测试门禁验证新旧路径的一致性。
- 任何一步出问题都可以回退，不影响系统运行。

## 四、功能需求（按层次）

### 4.1 意图策略层（IntentPolicy）

- 系统能接受到用户输入后，结合当前对话上下文、可用工具列表和历史偏好，生成一个结构化的执行决策。
- 决策内容包括：走哪种运行模板（默认对话 / 知识问答 / 单工具调用 / 多步工作流 / 文档处理）、允许使用哪些工具、是否启用记忆读写、是否需要先向用户澄清意图。
- 当系统不确定用户的意图时，优先请求澄清而不是强行猜测。
- 原先的关键词路由逻辑（检测是否要 RAG、是否要工具、是否要 ReAct）被新的决策层吸收，不再以散落的 if/else 形式存在。

### 4.2 工具边界层（MCP 封装）

- 搜索网页内容的能力通过统一工具暴露，后端不再依赖任何特定服务商。
- RAG 检索和记忆读写通过标准工具接口暴露，输入输出格式稳定、可文档化。
- 文档解析和入库通过标准工具接口暴露，后续替换解析引擎不影响调用方。
- 基础能力（时间、天气、命令执行）保持对外接口不变，内部实现可替换。
- 新增外部工具时，只需注册到工具边界层，编排引擎和子代理自动可见。

### 4.3 子代理层（AgentTeam）

- 保留四个预制子代理：资料搜集（research）、文档问答（doc_qa）、结果汇总（synthesis）、辅助执行（ops）。
- 每个子代理有明确的输入输出定义、工具权限范围和记忆策略。
- 支持新增自定义子代理，注册后即可被编排引擎调度。
- 子代理不直接访问底层实现，只能通过标准工具接口消费能力。

### 4.4 编排引擎（LangGraph）

- 每次用户请求对应一个执行图（graph），由编排引擎管理完整的生命周期。
- 支持基于意图决策进行条件路由——对话走轻量路径，多步任务走完整图执行。
- 长任务支持中断和恢复：中断时保存完整的执行状态，恢复时从断点继续。
- 支持并行执行：一个任务的多个独立子步骤可以同时进行。
- 记忆写入、事件通知等后置处理作为图中的固定节点执行，不再散落在各处。

### 4.5 配置与启动

- 配置有明确分层：默认值 → 共享配置 → 本机覆盖 → 环境变量 → 启动参数，覆盖顺序可解释。
- 密钥和密码只存在于环境变量或本机覆盖文件中，不出现在版本控制的配置里。
- 基础设施（数据库、搜索引擎等）初始化失败时能按能力降级，不阻塞无关功能的启动。
- 启动过程分阶段执行（加载配置 → 初始化基础设施 → 挂载工具 → 装配上下文 → 注册子代理 → 启动 Web 服务），每个阶段的失败原因可明确诊断。

### 4.6 运行时治理

- 可以统一查看当前任务的执行状态。
- 支持对运行中的任务进行取消。
- 任务的业务快照和执行状态（checkpoint）分开管理。

## 五、非功能性需求

- **兼容性**：重构过程中，HTTP API 的对外契约和前端交互界面保持不变。
- **测试门禁**：每个迁移步骤都有对应的测试保护；关键用户路径有回归测试覆盖。
- **错误处理**：系统任何一部分出错都不应导致整个服务不可用；面向用户返回安全的信息，内部保留足够的诊断上下文。
- **代码质量**：重构后模块职责单一、文件规模可控（单文件不超过 800 行）、命名清晰一致。

---

## 六、明确的非目标（不会在这次重构中做）

- 不会推翻重写 RAG、记忆、文档解析的底层算法和存储逻辑——它们被保留和复用，只会在外部加一层标准接口。
- 不会修改 HTTP API 的对外契约和前端交互行为。
- 不会引入新的基础设施依赖（数据库、消息队列等保持现有选型）。
- 不会一次性大规模重写业务逻辑——Phase 1 只是把边界划清楚，真正的实现迁移在后续阶段逐步完成。

---

## 七、成功标准

重构是否成功，不是看"写了多少新代码"，而是看以下几点：

1. **每一个现有能力都有明确的归属**：能说清楚 RAG 归谁管、记忆归谁管、工具归谁管、子代理归谁管。
2. **上层只依赖接口，不依赖实现**：编排引擎和子代理调用的是标准工具接口，不是底层类的具体方法。
3. **子代理可以独立扩展**：新增一个角色不需要改动主流程代码。
4. **配置安全**：默认仓库中不再包含任何真实凭据。
5. **旧功能继续可用**：聊天、RAG、工具调用、文档处理四种模式在重构后行为不受影响。
6. **每一步可独立验证、可回退**：不会出现"改到一半发现不对但已经回不去了"的情况。
