# ADR-007: 采用分层且不可变的配置装配

## 状态

已接受

## 背景

运行配置同时包含安全默认值、共享配置、本机秘密和部署覆盖。当前 tracked YAML 中存在默认凭据，且配置来源、未知字段处理和脱敏边界需要明确。

## 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| 继续直接读取单个 tracked YAML | 使用简单 | 凭据容易提交，覆盖和校验边界不清晰 |
| 所有配置只从环境变量读取 | 部署常见 | 本地复杂配置可读性差，无法复用现有 YAML contract |
| 代码默认值 → 共享 YAML → ignored local YAML → 白名单环境/启动覆盖 | 兼顾本地和部署，秘密不入库 | 需要深度合并、schema 校验和脱敏实现 |

## 结论

采用五层配置模型。mapping 深度合并，list 整体替换；未知字段、类型错误和非法组合启动失败；装配完成后配置只读。真实 API key、密码和 token 只允许来自 ignored local 配置、受控外部 YAML 或后续明确的 secret reference，不写入 tracked 文件、日志和用户错误响应。

## 后果

- `config.yaml` 只能作为无秘密模板，Compose 默认凭据必须改为外部注入。
- Phase 0 必须清理当前 tracked 默认凭据并核对 Git 历史；历史凭据是否重写需仓库所有者明确授权。
- Phase 1 才实现完整的不可变配置模型和覆盖测试。
