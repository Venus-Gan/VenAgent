# ADR-013: 采用 canonical `apps`/`src` 布局并退役 `final`

## 状态

已接受

## 背景

当前实现集中在 `final/`，其中包含运行入口、领域能力、编排、适配器和测试边界。继续向 `final/internal` 堆叠新代码会延长耦合；直接整树搬迁又缺少测试保护。

## 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| 继续扩展 `final/internal` | 改动最少 | 永久兼容壳化，边界继续恶化 |
| 立即整树搬迁并删除 `final` | 目标清晰 | 回归和回滚风险过高 |
| 分阶段建立 canonical 目录，`final` 单向委托 | 可测试、可回滚，逐步收缩旧入口 | 迁移期需要维护兼容入口 |

## 结论

目标布局为 `apps/`、`src/venagent/`、`tests/`、`config/` 和 `deploy/`。Phase 0 仍以 `final/` 建立行为与安全基线；Phase 1 起建立新包，禁止新包反向导入 `final`；稳定发布窗口后删除 `final`。

## 后果

- 目录迁移必须按 Phase 进行，不做无测试保护的整树搬家。
- 旧入口只允许作为兼容 façade 委托新应用，不能继续承载新领域逻辑。
- 删除 `final` 前必须通过引用扫描、全量测试、E2E、性能和回滚门禁。
