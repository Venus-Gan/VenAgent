# Phase1.重构计划

## 目标场景
你现在要把旧项目重构成 `VenAgent`(/home/ubuntu/VenAgent) 的第一阶段 SDD 方案。

这一阶段不追求一次性重写完，而是先把底层能力、编排层、工具边界、子 agent 扩展方式全部定清楚，确保后续实现能渐进迁移、可回滚、可扩展。

## 场景化目标

### 场景 A：你还想保留原来的底层能力
RAG、memory、文档解析、索引、召回、持久化这些核心逻辑继续保留在项目内部，不拆散、不推倒重来。

### 场景 B：你希望未来所有能力都能被更标准地调用
对外统一通过 MCP tool 暴露，让 LangGraph 和后续 agentteam 只接触标准化工具接口。

### 场景 C：你希望有可扩展的子 agent 体系
新增 `agentteam` 作为项目内模块，负责子 agent 编排。
原先 RAG 里钉死的四个 agent 迁移为 `agentteam/presets`，作为预制子 agent 交给后续工作流复用。

### 场景 D：你希望迁移过程中风险可控
Phase 1 只做架构边界、模块分层、接口定义和迁移路径设计，不做大规模逻辑重写。

## Phase 1 技术决定

### 已确认技术栈
- 编排层：`LangGraph`
- 适配层：`LangChain`
- 工具边界：`MCP`
- 子 agent 组织层：`agentteam`
- Web 抓取与网页内容获取：`fetch MCP`
- RAG 与 memory：保留底层实现，外层封装为 MCP tool
- `search_web` 不再依赖 Tavily，改由 `fetch MCP` 作为网页内容获取后端

### 设计原则
- 保留现有底层资产
- 新增标准接口层
- 编排与工具解耦
- 子 agent 配置化、预制化、可扩展
- 先结构化迁移，再逐步替换实现
- 内部中间表示统一命名为 `WorkflowSpec`，使用符合代码规范的 PascalCase 命名，不再使用临时占位名

## Phase 1 快速判断
| 问题 | 结论 |
|---|---|
| 哪些能力保留在底层 | RAG、memory、document、索引、知识图谱、恢复态 |
| 哪些能力对外统一暴露 | 时间、天气、搜索、命令执行、文档、RAG、memory |
| 哪些能力交给 LangGraph | 路由、状态、并行、子图、恢复、后置副作用 |
| 哪些能力交给 agentteam | 子 agent 注册、选择、执行、预制与扩展 |
| 哪些能力先不动 | HTTP API 和前端交互层 |

## 验收口径
Phase 1 不是实现完，而是把后续实现的边界定住。

当以下条件成立时，Phase 1 就算进入可执行状态：
- 每个现有能力都有明确归属
- `Core / MCP / Graph / AgentTeam` 边界清晰
- 原固定 RAG agent 有落点
- RAG / memory 作为底层资产被保留
- 对外工具接口统一为 MCP
- 后续实现可以按模块逐步推进
