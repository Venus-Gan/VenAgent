# Phase1.核心特色与边界

## 目的
把重构后必须继续保留的项目特色和不能被 MCP 吞掉的边界单独列出来，避免后续实现时把控制面也一并工具化。

---

## 核心特色保留
| 特色 | 保留方式 | 关键约束 |
|---|---|---|---|
| ReAct 任务图执行 | Planner 输出 `WorkflowSpec`，Graph Runtime 执行 | 能拆步、能依赖、能重试 |
| 动态生成结构化任务图 | 不写死固定 pipeline | 每个任务可生成不同拓扑 |
| Subagent 子代理流水线 | `agentteam` 承担子代理组织 | 子代理可接力、可替换、可扩展 |
| 失败重试与长任务恢复 | checkpoint + PostgreSQL 快照 | 可中断、可恢复、可局部修复 |
| 任务图与子代理的关系 | 图管顺序，子代理管执行 | 控制面与执行面分离 |

### 1. ReAct 任务图执行
| 关键点 | 说明 |
|---|---|
| 目标 | Planner LLM 根据用户意图和可用工具生成结构化任务图 |
| 保留内容 | 分析任务、拆分步骤、形成依赖关系、按拓扑顺序执行 |
| 结果要求 | 失败节点可以进入重试或回退流程 |

### 2. 动态生成结构化任务图
| 关键点 | 说明 |
|---|---|
| 核心要求 | 不把执行流程写死成固定 pipeline |
| 核心表示 | `WorkflowSpec` 作为中间表示 |
| 执行方式 | Runtime 编译 spec，再执行 graph |

### 3. Subagent 子代理流水线
| 关键点 | 说明 |
|---|---|
| 组织方式 | `agentteam` |
| 预制 agent | 原来固定的四个 agent 迁入 presets |
| 扩展方式 | 自定义 agent 可插入新流程 |

### 4. 失败重试与长任务恢复
| 关键点 | 说明 |
|---|---|
| 恢复机制 | LangGraph checkpoint 保存状态 |
| 持久化 | 快照进 PostgreSQL |
| 恢复能力 | 支持中断后恢复、回放、局部修复 |

### 5. 任务图与子代理的关系
| 组件 | 负责 |
|---|---|
| 任务图 | 要做什么、按什么顺序做 |
| 子代理 | 由谁做、如何做 |
| MCP | 能力面 |
| RAG / memory | 底层能力面 |

### 不能被 MCP 化吞掉的部分
| 内容 | 原因 |
|---|---|
| Planner 生成任务图的逻辑 | 这是控制面，不是单纯能力调用 |
| 图的编译与执行语义 | 关乎任务调度和状态迁移 |
| 节点之间的依赖管理 | 关系表达不能丢 |
| 并行层级调度 | 不是普通 tool 能替代的 |
| checkpoint / resume 机制 | 是长任务稳定性的核心 |
| 子代理流水线编排 | 是整个 agent 协作骨架 |

---

## 目标架构
| 层级 | 角色 | 归属 |
|---|---|---|
| 核心层 | RAG、memory、document、retrieval、indexing | Core |
| MCP 适配层 | 将核心能力包装为工具 | MCP |
| 编排层 | 路由、任务分发、子流程、状态流转、恢复 | LangGraph |
| 子 agent 层 | 预制子 agent、自定义子 agent、角色协作 | agentteam |

---

## 迁移边界
| 类别 | 内容 |
|---|---|
| 保留 | RAG 底层召回、memory 底层读写、文档解析与索引底座、现有业务规则 |
| 重构 | 工具暴露方式、agent 编排方式、路由与任务分发方式、预制 agent 组织结构 |
| 新增 | `agentteam` 模块、MCP 工具封装层、LangGraph 编排入口、统一状态与契约定义 |

---

## 验收标准
- 项目特色没有被工具化稀释
- 控制面、执行面、能力面边界明确
- 后续实现不会把任务图、子代理、恢复机制再次散落到业务函数里
