# Phase1.IntentPolicy

## 定位
`IntentPolicy` 是主 agent 和 LangGraph runtime 之间的轻量决策层。

它负责把“用户意图 + 当前上下文 + 能力边界”翻译成“这次请求应该如何运行”。

它不是具体能力实现，也不是工具层、RAG 层或子 agent 层，而是一个稳定、可版本化、可回退的运行策略层。

---

## 为什么要单独抽出来
LangGraph 官方建议把应用拆成离散节点，再通过共享 state、conditional edges 和 subgraph 串起来；LangChain 也推荐用结构化输出而不是自由文本解析来返回可预测结果。基于这两点，`IntentPolicy` 应该从主 agent 里独立出来，而不是继续藏在 `UnifiedAgent._prepare()` 的大 if/else 里。

---

## 核心职责
| 职责 | 说明 |
|---|---|
| 意图收敛 | 把用户输入和当前会话上下文收敛成可执行决策 |
| profile 选择 | 选择本次请求应该采用的运行模板 |
| promptctx 策略 | 决定装配哪些上下文槽位、预算和裁剪规则 |
| 能力边界 | 决定允许哪些 MCP tool、子代理和恢复能力 |
| 路径选择 | 决定走轻量回答、工具调用、图执行还是子图 |
| 回退策略 | 决定何时降级到默认对话或澄清请求 |
| 兼容映射 | 把旧 `chat / rag / tool / react` 映射成新 profile |

---

## 推荐输出契约
建议 `IntentPolicy` 输出一个版本化、结构化的决策对象，而不是返回一段自由文本。

### `IntentDecision` 建议字段
| 字段 | 说明 |
|---|---|
| `policy_version` | 决策版本，方便以后升级兼容 |
| `intent_class` | 用户意图分类，如 `conversation` / `retrieval` / `tool_call` / `workflow` |
| `execution_profile` | 运行模板名，如 `default_chat` / `knowledge_answer` / `single_tool` / `workflow_task` |
| `prompt_schema_key` | `promptctx` 使用的 schema 名 |
| `graph_entry` | LangGraph 起始节点或子图入口 |
| `tool_scope` | 本次允许的工具范围 |
| `agent_scope` | 本次允许的 agentteam 角色或 preset |
| `memory_scope` | 是否读写 STM / LTM / preference / task memory |
| `recovery_policy` | 是否启用 checkpoint、resume、interrupt |
| `clarify_needed` | 是否需要澄清用户意图 |
| `confidence` | 该决策的置信度 |
| `reason` | 简短可审计原因 |

---

## 输入来源
| 输入 | 说明 |
|---|---|
| 用户输入 | 当前自然语言请求 |
| 对话状态 | STM、最近上下文、历史结果 |
| 工具快照 | 当前可用工具、MCP 能力、沙箱可用性 |
| 记忆信号 | 长期偏好、会话记忆、任务记忆 |
| 配置开关 | 例如是否允许高风险操作、是否启用某些 profile |
| 用户显式选择 | 例如显式选工具、显式 use_rag、显式 mode |
| 安全约束 | 允许列表、命令限制、执行边界 |

---

## 输出 profile 建议
| profile | 含义 | 兼容旧概念 |
|---|---|---|
| `default_chat` | 默认对话型运行模板 | `chat` |
| `knowledge_answer` | 检索增强型运行模板 | `rag` |
| `single_tool` | 单工具执行型运行模板 | `tool` |
| `workflow_task` | 图执行/多步任务运行模板 | `react` |
| `document_flow` | 文档型工作流模板 | 旧文档任务路径 |
| `clarify` | 需要澄清后再执行 | 新增 |

说明：旧四模式可以继续保留为兼容标签，但它们不应该再作为主分层骨架。

---

## 推荐决策流程
### 1. 先做确定性预过滤
优先处理这些显式信号：
- 用户明确指定工具或模式
- `selected_tools` 不为空
- 明确的文档工作流请求
- 明确的命令类请求
- 显式要求走 RAG 或不走 RAG

### 2. 再做结构化分类
当显式规则不足时，再调用结构化分类器，把请求归入标准意图类别。

这里建议使用 LangChain 的结构化输出能力，让分类结果直接落到 Pydantic / dataclass / JSON schema，而不是让模型吐自由文本再手工解析。

### 3. 最后做归一化
把分类结果归一化成：
- `execution_profile`
- `prompt_schema_key`
- `graph_entry`
- `tool_scope`
- `agent_scope`
- `memory_scope`

### 4. 低置信度时不要硬猜
如果策略判断不稳，优先返回 `clarify`，而不是把不确定请求硬塞进错误的 profile。

## 与 LangGraph 的关系
| 组件 | 负责 |
|---|---|
| `IntentPolicy` | 生成本次请求的策略决策 |
| `State` | 承载当前请求的共享状态 |
| `START` + conditional edges | 按策略决定进入哪个节点或子图 |
| `Send` | 后续需要并行 fan-out 时再使用 |
| checkpoint/store | 保存线程状态和跨线程记忆 |

LangGraph 官方文档明确支持从 `START` 通过 `add_conditional_edges` 做条件入口，也支持基于 state 的条件路由、`Send` 风格的 map-reduce，并把 checkpointer / store 分别用于线程级状态和跨线程持久化。`IntentPolicy` 正好应该生成这些路由所需的结构化输入。

---

## 与主 agent 的关系
| 组件 | 负责 |
|---|---|
| `IntentPolicy` | 决定本次请求如何跑 |
| `UnifiedAgent` | 调用 `IntentPolicy`，然后把结果交给 LangGraph runtime |
| `promptctx` | 根据 `prompt_schema_key` 装配上下文 |
| `agentteam` | 根据 `agent_scope` 执行子 agent 或 preset |
| `MCP tool` | 根据 `tool_scope` 提供标准能力 |

---

## 与当前代码的对齐
| 当前代码 | 现在的职责 | 重构后归属 |
|---|---|---|
| `router.py` | `need_tool / need_rag / need_react` 关键词判断 | `IntentPolicy` 的特征输入来源 |
| `agent.py._prepare()` | 决定 mode、装配上下文、过滤工具 | 调用 `IntentPolicy.resolve()` |
| `promptctx.schema.py` | 定义 chat/tool/react/rag 的槽位模板 | 由 `prompt_schema_key` 驱动 |
| `graph_runtime.py` | 执行拓扑、竞速、重试、取消 | 由 `graph_entry` 和 profile 触发 |
| `memory_writer.py` | 记忆抽取与合并 | 由 `memory_scope` 决定是否启用 |

---

## 实现建议
建议把这层做成轻量模块，而不是又一个大而全的中间层。

可选目录结构：
```text
internal/agent/policy/
  intent_policy.py
  types.py
  registry.py
```

推荐职责：
| 文件 | 职责 |
|---|---|
| `types.py` | `IntentSignal`、`IntentDecision`、`ExecutionProfile` |
| `intent_policy.py` | 规则预过滤 + 结构化分类 + 决策归一化 |
| `registry.py` | profile 到 schema / graph / capability 的映射 |

---

## 这层不负责什么
| 不负责 | 原因 |
|---|---|
| 工具实现 | 工具已经收口到 MCP tool |
| RAG 实现 | RAG 是底层能力，不是策略层职责 |
| 子 agent 实现 | 子 agent 由 agentteam 负责 |
| 任务图执行 | 由 LangGraph / GraphRuntime 负责 |
| 配置加载 | 由 bootstrap / config 负责 |

---

## 对 Phase 1 的补充要求
| 需要补的点 | 原因 |
|---|---|
| `IntentPolicy` 独立成文 | 避免把策略和主 agent 混在一起 |
| 旧四模式降级为兼容标签 | 保留兼容性，但不主导架构 |
| 用结构化输出代替自由文本解析 | 降低解析失败和语义漂移 |
| 明确策略层与 LangGraph 的边界 | 后续实现才不会重回大 if/else |
| 低置信度时可澄清 | 稳定性优先于盲猜 |

---

## 验收标准
- 能解释一次请求如何从输入进入 `IntentPolicy`
- 能解释 `IntentDecision` 如何驱动 `promptctx` 和 `LangGraph`
- `chat / rag / tool / react` 不再是主分层，只是兼容标签
- 策略层输出是结构化的、可审计的、可回退的
