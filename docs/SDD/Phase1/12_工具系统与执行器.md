# Phase1.工具系统与执行器

## 定位
当前代码里的工具系统不只是“若干工具函数”，而是一整套：工具模型、注册表、搜索降级链、文档工具、沙箱工具和 MCP 风格动态工具的组合。

这部分在 Phase 1 里需要单独看清楚，因为它直接决定 MCP 化边界和搜索后端替换方式。

---

## 当前工具栈
| 层级 | 代码位置 | 作用 |
|---|---|---|
| 工具模型 | `final/internal/tools/tools.py` | `Tool` / `CallResult` / 默认工具集 |
| 工具注册表 | `ToolExecutor` | 注册、覆盖、查询、执行 |
| 搜索实现 | `search_web_factory()` / `tavily.py` | Tavily → LLM → mock |
| 沙箱执行 | `exec_command.py` | 受控命令执行 |
| 文档工具 | `UnifiedAgent._register_document_tools()` | `write/read/list/ingest` |
| RAG 工具 | `rag_search` 闭包 | 依赖主 agent 的 RAG 实例 |
| MCP 兼容工具 | `new_mcp_tool()` / `register_mcp_tool()` | 动态挂载外部能力 |

---

## 默认工具现状
| 工具 | 当前行为 | 计划中的目标 |
|---|---|---|
| `get_time` | 返回当前时间 | 保留，封装为 MCP tool |
| `get_weather` | 内置模拟天气 | 保留，封装为 MCP tool |

| `write_document` | 写文档并可同步入 RAG | 保留，拆成文档服务与 MCP tool |

## 现在的搜索链路
`search_web_factory()` 当前逻辑是：

1. 有 `search_api_key` 时，优先走 Tavily 真实搜索
2. Tavily 失败时，降级到 LLM 生成回答
3. LLM 也不可用时，回退到 mock 搜索

这意味着当前项目里，搜索能力仍然是“API key 驱动”的，而不是“统一 MCP tool 驱动”的。

### Phase 1 目标
| 项目 | 目标 |
|---|---|
| 搜索后端 | 改成 `fetch MCP` |
| 配置项 | 删除 `search_api_key` / `search_api_url` |
| 工具形态 | 只保留统一的 `search_web` MCP tool |
| 兼容策略 | 迁移期可保留适配层，但最终不再依赖 Tavily |

## 工具注册表语义
| 方法 | 语义 | 对 Phase 1 的影响 |
|---|---|---|
| `call()` | 取工具、执行、返回 `CallResult` | 需要保留结构化执行结果 |
| `add_tool()` | 覆盖同名工具 | 适合 MCP 动态挂载 |
| `snapshot()` | 读出当前工具快照 | 适合 Planner / 路由 / promptctx |
| `filter_tools()` | 按名单取子集 | 适合显式指定工具集 |
| `get_tool_descriptions()` | 暴露工具 schema | 适合前端、Planner、LLM 发现 |

## MCP 化边界
| 现有能力 | 当前问题 | Phase 1 目标 |
|---|---|---|
| `get_time` / `get_weather` | 只是普通函数工具 | MCP tool |
| `search_web` | 内部 provider chain 太重 | MCP tool + fetch backend |
| `exec_command` | 与 sandbox 绑定 | 独立 sandbox 边界 + tool wrapper |
| `rag_search` | 直接闭包到主 agent | 封装成 RAG MCP tool |
| 文档工具 | 直接绑定 `UnifiedAgent` | 独立 `DocumentService` + MCP tool |

---

## 需要单独保留的边界
### 1. 沙箱不是普通工具
沙箱是受控执行边界，不是普通业务工具集合的一部分。

### 2. 文档工具不是 RAG 本体
`write/read/list/ingest` 是文档库链路，RAG 只是其中一个消费方。

### 3. 搜索后端不能继续散落
Tavily 当前还在代码里，Phase 1 目标是把它从 config 和默认工具链里清掉，改由 `fetch MCP` 承接。

---

## 对 Phase 1 的补充要求
| 需要补的点 | 原因 |
|---|---|
| 工具执行器单独成文 | 这是 MCP 化的真正落点之一 |
| 搜索后端替换写清楚 | 现在代码和既定方案不一致 |
| 文档工具单独列出 | 它和 RAG 相关，但不等于 RAG |
| 沙箱边界继续独立 | 不能被普通 tool 概念吞掉 |

---

## 验收标准
- 能清楚区分工具、沙箱、文档库、RAG 的边界
- `search_web` 的后端替换路线明确
- `ToolExecutor` 的注册和执行规则可复用到未来的 MCP 封装
